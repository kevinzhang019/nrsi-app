## MLB Stats API gotchas

- **Live feed lives at `/api/v1.1/...`**, not `/api/v1/...`. v1 returns 404 for the same path.
- **Split sitCodes are `vl,vr`**, NOT `vsl,vsr`. The wrong codes return an empty `splits[]` array — silent failure, hardest kind of bug.
- **Splits don't exist for players with no PAs in that split this season.** Legacy v1 (`loadBatterProfile`/`loadPitcherProfile`) falls back to prior season (`SEASON - 1`) only when `stats[0].splits` is empty. v2 (`loadBatterPaProfile`/`loadPitcherPaProfile`) goes further: always fetches **both** `SEASON` and `SEASON - 1` in parallel and PA-weighted-blends them with a role-specific Marcel-style multiplier on the rates (hitter `3:2`, pitcher `2:1`), then EB-shrinks the combined baseline against true PA (hitter `n0 = 200`, pitcher `n0 = 500`) and folds in a 10% last-30-day blend when L30 has ≥ 80 PA. `season=YYYY` aggregates exclude postseason for free.
- **Switch-hitter platoon resolution defaults to canonical** (`actual` rule — bat opposite of pitcher hand). Legacy `max(L, R)` reachable via `NRXI_SWITCH_HITTER_RULE=max`. Implementation in `lib/prob/log5.ts:effectiveBatterStance` and `batterSideVs`.
- **`/people/{id}.pitchHand.code` returns `"S"` for switch-throwers** (rare; usually position players who pitch in mop-up duty, e.g. Carlos Cortes). Schema parsing uses `PitchHandRaw` (`"L"|"R"|"S"`) at the wire boundary; downstream code uses `PitchHand` (`"L"|"R"`) after `lib/mlb/splits.ts:resolveThrowsHand` collapses `"S"` by picking the lower-WHIP side from the player's pitching splits (defaults to `"R"` when no stats). Tightening the wire schema back to `"L"|"R"` re-fires BUGS.md bug #13.
- **`boxscore.teams.*.battingOrder` is empty until lineups post** (~30 min before first pitch). `getUpcomingForCurrentInning` returns `null` if the array is < 9 long.
- **`outs === 3` flickers at half-inning transitions.** Don't use raw `outs` as the recompute trigger; use a composite `inningKey = "${inning}-${half}-${outs >= 3 ? 'end' : inningState || 'live'}"`.
- **Respect `metaData.wait`.** The live feed includes a server-side hint (typically 10s). Polling faster wastes calls and risks rate limits.
- **`User-Agent` matters.** Set `MLB_USER_AGENT` env var to identify yourself; the default is `nrxi-app/0.1`.
- **`/people/{id}` carries `currentAge` and `birthDate`** — both optional in our zod schema (`lib/mlb/types.ts:PersonResponse`). `loadHand` prefers `currentAge` and falls back to `ageFromBirthDate(birthDate)`. Used by `lib/mlb/aging.ts:applyAging` to project the prior-season rates forward 1 year before blending. Missing age silently degrades to identity (no projection); don't gate the splits fetch on age availability.
- **`stats=byDateRange&group=pitching`** returns aggregated `numberOfPitches` across the window — used by `lib/env/workload.ts:loadRecentPitchCount` for the 7-day pitch total. Different from `byDateRangeSplits` (which is split by L/R but doesn't surface pitch count). Cached 6h per player.
- **Savant park-factor leaderboard accepts `batSide=L|R`** for handedness-controlled factors. `lib/env/park.ts:getParkComponentFactors` now fetches three tables in parallel (combined + L + R) and populates `ParkComponentFactors.L/R` distinctly. The empty `batSide=` query returns the combined index. Used as the fallback when either side's scrape comes back empty.
- **Savant `expected_statistics` leaderboard** (`?type=batter&year=YYYY&min=10`) carries `xhr`, `pa`, `bbe`, `hr`, `xwoba`, `brl_percent`. Used to denoise hitter HR rates in `lib/env/expected-stats.ts:hrRateMultiplier`. Pitcher xstats deliberately not used (BP "Siren Song").
- **FanGraphs Stuff+/Pitching+ leaderboard** (`/api/leaders/major-league/data?stats=pit&type=36&season=YYYY`) carries `Stuff+`, `Location+`, `Pitching+`. Join to MLB Stats API ids via the `xmlbamid` field — when missing, the row is unjoinable and the model degrades to identity. FG URLs are less stable than Savant; treat the scrape as best-effort.

## Railway watcher conventions

The watcher used to run on Vercel Workflow DevKit; that's been replaced by a Railway-hosted Node supervisor. Detail in `docs/ARCHITECTURE.md`; ops in `docs/RUNBOOK.md`. Invariants:

- **Cron:** `0 12 * * *` UTC declared in `railway.toml`. Entry point `bin/supervisor.ts` → `services/supervisor.ts:runSupervisor`. Restart policy `NEVER` so the supervisor's clean exit scales the container to zero.
- **Pre-game lead 6h.** Each watcher task scheduled at `gameDate − 6h` via `setTimeout`, which idles in setTimeout (zero Redis / MLB / lock cost) until the spawn time arrives. Day games (≤ 18:00 UTC) hit `Math.max(now, gameDate - 6h)` resolving to supervisor wake; night games spawn later in the day. 6h is the conservative tail of when lineups actually post (typically T-1h to T-3h) and when probable pitchers are locked in (usually by T-6h); going longer just burns Redis lock-refresh ops (10s cadence × pre-game hours × every game) and MLB feed polls for no signal. Pre-game cadence inside the watcher is 30 min. The watcher computes first-inning predictions during pre-game whenever lineups + probable pitchers are available; persistence to `inning_predictions` is gated on `status === "Live"` so only the live-tick recompute (= prediction at inning-actually-begins) hits Supabase. See "Pre-game compute" invariant under Don't change without thinking.
- **Idle-exit predicate:** `pending.size === 0 && Date.now() >= tomorrow06UTC`. Off-season days exit within seconds. The 06:00 UTC cutoff protects late-running doubleheaders.
- **Single-instance lock:** every watcher acquires `nrxi:lock:{gamePk}` with a **30s TTL** via `services/lib/lock.ts:acquireWatcherLock`. A background `setInterval` refreshes every 10s in `startLockRefresher` — decoupled from loop cadence so long pre-game / Delayed sleeps don't risk lock expiration.
- **State durability:** hoisted vars (`lastNrXi`, `lastEnv`, last-pitcher cores, etc.) JSON-serialized to `nrxi:watcher-state:{gamePk}` on every tick (`services/lib/watcher-state.ts:saveWatcherState`). On watcher restart, hydrated by `loadWatcherState`. Trigger keys are deliberately NOT persisted — first tick after restart unconditionally fires Phase 1 reload to rebuild caches. **`capturedInnings` is intentionally NOT in the saved bundle** — predictions are durable in Supabase from the moment they're computed via per-boundary `upsertInningPrediction`; the in-memory map is only a thin retry/dedup buffer.
- **Snapshot pruning:** supervisor calls `services/lib/prune-snapshots.ts:pruneStaleSnapshots({ todayET })` after seedSnapshot to delete any `nrxi:snapshot` field-keys whose row's own `officialDate < todayET`. Required because `publishGameState` resets the hash's 24h TTL on every tick — without prune, prior-day games linger forever. Discriminator is the row's own date (not the schedule fetch's pk list) so a partial/empty schedule fetch on a rerun can't wipe still-scheduled games. See BUGS.md bug #10.
- **Stale-Live detector:** supervisor calls `services/lib/stale-live-detector.ts:detectAndCleanStaleLive` every `IDLE_CHECK_INTERVAL_MS` (60s) inside the idle loop. Scans `nrxi:snapshot` for entries where `status === "Live"` AND lock missing AND `updatedAt > 60s ago` and republishes a synthetic `{ ...lastState, status: "Final" }` so the dashboard moves the game out of Active. The lock-missing check is the load-bearing one (the `startLockRefresher` keeps the lock fresh every 10s while a watcher is alive); the 60s threshold is just a publish/exit race-condition margin. This is the only defense against process-kill scenarios (SIGKILL, OOM, container eviction) where the in-process gracefulExit path can't run. See BUGS.md bug #11.
- **Sweep-finalize:** supervisor also calls `services/lib/sweep-finalize.ts:sweepFinalize` once per idle iteration (and once at supervisor exit). For every today-bucket game whose archive isn't fully written (stub-only `games` row, missing `linescore`, or any `inning_predictions.actual_runs` still NULL), it does one `fetchLiveFull` and — if MLB has flipped to Final — runs `finalizeGame` (UPSERT `games` + UPSERT `plays` + UPDATE `actual_runs` from `linescore.innings[]`). Idempotent. This is the centralized post-game persistence path; the watcher itself does no DB writes at Final. Independent of `detectAndCleanStaleLive`, which only touches the dashboard snapshot. See BUGS.md bug #12.
- **Adaptive sleep:** `services/steps/fetch-live-diff.ts:chooseRecommendedWaitSeconds` governs Live cadence only — **5s** during active PAs (Live + `inningState ∉ {Middle, End}` + `outs < 3`), **15s** otherwise (inning breaks, pitching changes, replay reviews). Pre-game and Delayed/Suspended cadences are hard-coded in `services/run-watcher.ts`'s sleep switch: **30min** pre-game (was 30s before the pre-game-compute feature; the longer window is fine because the structural-key change detector catches any lineup/pitcher diff on the next tick), **5min** Delayed/Suspended.
- **Hard runtime caps:** `MAX_LOOPS = 5000` (≈7h at the 5s active cadence) and `MAX_RUNTIME_MS = 20h`. Realistic max watcher lifetime under `PRE_GAME_LEAD_MS = 6h` is ~11h (6h pre-game + 5h game with extras). The 20h cap is defense-only — the supervisor's own idle-exit deadline (06:00 UTC next day = ~18h after wake) is the practical ceiling. MAX_LOOPS stays at 5000 because pre-game ticks at 1800s add only ~12 iterations across the 6h window. Historically MAX_LOOPS was 1500 (≈2h) and watchers hit it during the late innings of normal games — see BUGS.md bug #11. Don't drop `MAX_RUNTIME_MS` back below the supervisor's ~18h lifetime without also tightening the pre-game lead — under any wider lead, the cap would fire pre-first-pitch and `gracefulExit` would flip scheduled games to synthetic Final.
- **Graceful non-Final exit:** every watcher exit path that isn't the Final branch (`max-loops`, `max-runtime`, `abort` SIGTERM, `error` uncaught throw) calls `services/lib/finalize-game.ts:performGracefulExit` with the last successfully-published state. **When `lastPublishedState.status === "Live"`**, the helper publishes `{ ...lastState, status: "Final" }` so the dashboard moves the game out of Active, then clears the `nrxi:watcher-state:{gamePk}` key. **When `lastPublishedState.status === "Pre"`** (a watcher dying mid-pre-game — under `PRE_GAME_LEAD_MS = 6h` this is most realistically an `abort` from Railway redeploy or an `error` from an uncaught throw, since `max-runtime` no longer fires before first pitch), the synthetic Final publish is **skipped** — flipping an Upcoming card to Finished hours before first pitch is the wrong UI signal. The watcher-state is still cleared so the next supervisor cron's fresh watcher rebuilds caches from scratch. **No DB writes** in either branch — predictions are already in Supabase via the watcher's per-boundary `upsertInningPrediction` calls, and the supervisor's `sweepFinalize` handles `games`/`plays`/`actual_runs` from a fresh `fetchLiveFull` once MLB flips the game to Final. Best-effort — never throws. The main loop body in `services/run-watcher.ts` is wrapped in a try/catch that routes any uncaught step error into `gracefulExit("error")` instead of bubbling to the supervisor. See BUGS.md bug #11 (stuck-Live cleanup) and bug #12 (per-inning persistence).
- **Per-boundary prediction persistence:** at every clean half-inning boundary (`buildInningCapture` returns non-null), the watcher fires-and-forget `upsertInningPrediction` to Supabase. Failures are logged but never block the tick — the supervisor's `sweepFinalize` backstops any rows that don't make it. The first such call also lazily inserts a stub `games` row (`ON CONFLICT DO NOTHING`) to satisfy the FK; the sweep upgrades it to a full row at finalization. See BUGS.md bug #12.
- **Logging:** plain `console.log` is fine — Railway captures stdout/stderr. `lib/log.ts` emits structured JSON `{t, level, scope, msg, ...}` filterable in the Railway dashboard.

## Persistence (Supabase)

Permanent archive of finished games lives in Supabase Postgres (free tier via Vercel Marketplace). Three tables, written by **two separate paths** — watcher per-boundary for predictions, supervisor sweep for everything else:
- `games` — one row per game; JSONB blobs for linescore / lineups / final_snapshot. Stub row inserted by the watcher's first `upsertInningPrediction` (lazy, `ON CONFLICT DO NOTHING`); upgraded to full final shape by `sweepFinalize`.
- `inning_predictions` — one row per (game_pk, inning, half); the model's prediction at the start of that half-inning. Written by the watcher fire-and-forget at each clean half-boundary. `actual_runs` filled by `sweepFinalize` from `linescore.innings[].{away,home}.runs`.
- `plays` — one row per completed plate appearance, keyed `(game_pk, at_bat_index)`. Powers per-inning hitter / pitcher rollups on the history detail page. Captured by `sweepFinalize` from a fresh `fetchLiveFull` once MLB flips the game to Final.

- **Schema:** `supabase/migrations/0001_history.sql` (base) + `0002_extras.sql` (relax `inning between 1 and 12` → `inning >= 1` for extras) + `0003_plays.sql` (per-PA archive). Apply via Supabase dashboard SQL editor or `node --env-file=.env.local scripts/migrate.mjs` (postgres-js, idempotent — every statement is `if not exists` or guarded). Service role is the only writer; RLS stays off.
- **Client:** `lib/db/supabase.ts` follows the `redisRestConfig()` / `redis()` split — `supabaseConfig()` for env-var lookup, `supabaseAdmin()` lazy singleton (service role).
- **Inning-prediction capture point:** `services/capture-inning.ts:buildInningCapture` is a pure helper called from `services/run-watcher.ts` after `state` is built and before `publishUpdateStep`. The **load-bearing guard** is `nrXi.startState.outs === 0 && (bases === 0 || bases === 2)` — that's true exactly at half-inning boundaries: `bases === 0` for regulation (1–9), `bases === 2` for the Manfred runner on 2B in extras (10+). Matches `readMarkovStartState`'s injection on `inningState=middle/end` / `outs >= 3`. The once-per-`${inning}-${half}` in-memory map guard prevents subsequent ticks from re-firing the same write.
- **Watcher write path:** `lib/db/inning-predictions.ts:upsertInningPrediction({context, capture})`. Two sequential supabase-js calls, both idempotent — `INSERT ... ON CONFLICT DO NOTHING` for the stub games row + `UPSERT` for the prediction row. Called fire-and-forget from the watcher's main loop body; logs on failure but never blocks the tick. No-ops when Supabase env vars are missing.
- **Sweep-finalize path:** `lib/db/inning-predictions.ts:finalizeGame({gamePk, freshFeed})`. UPSERT `games` (full final shape from feed) + UPSERT `plays` (built via `buildPlayRows(freshFeed, gamePk)`) + parallel UPDATE `inning_predictions.actual_runs` for each row in `linescore.innings[]`. Called by `services/lib/sweep-finalize.ts:sweepFinalize` from the supervisor's idle loop (every 60s) and at supervisor exit. Idempotent — re-running over an already-finalized game produces no diff.
- **Centralized post-game persistence:** the watcher's Final branch does NO DB writes — it just clears watcher state and exits. All `games`/`plays`/`actual_runs` work happens in `sweepFinalize`. Trade-off: ≤60s lag from MLB-Final → `/history` listing. The dashboard moves the game out of Active immediately via the watcher's last `publishUpdateStep` (already Final-status by then).
- **Per-inning rollups:** `lib/history/rollup-plays.ts:rollupBatters/rollupPitchers` are pure functions. Caller pre-filters `PlayRow[]` to the desired (inning, half) slice. `ipOuts` is computed by walking plays in `at_bat_index` order and tracking running outs per (inning, half), attributing each `max(0, endOuts − prevOuts)` increment to the pitcher who threw the play.
- **History bucket key:** `lib/db/games.ts:gameDateOf(officialDate, startTime)` prefers `GameState.officialDate` (venue-local YYYY-MM-DD from MLB's `gameData.datetime.officialDate`) over a TZ-converted derivation from `startTime`. Late-night PT games no longer slip into the next UTC day.
- **History UI:** `/history` (date strip + calendar popover, only data-days enabled) and `/history/[pk]` (single wide frozen `<GameCard wide>` whose `<LineScore>` cells are themselves the inning picker — click the inning *number* for full-inning composition, click the runs cell in away/home row for that half. State lives in `<HistoricalGameView>`; `<HistoricalPlaysPanel>` renders per-tab batter / pitcher tables + play log below). Both pages put `connection()` + `await params/searchParams` inside the `<Suspense>` body — never at the page level — to keep Cache Components happy.

## Default decisions worth preserving

- **v2.2 model is the default.** Probability pipeline is `Log5 → applyEnv → applyTtop → applyFraming → applyDefense → 24-state Markov → stratified calibrate`. Per-batter / per-pitcher profile loaders bake aging projection, Savant xHR denoiser (hitters), Stuff+ K↑/HR↓ bias (pitchers), 7-day workload drag (pitchers), and per-batter GIDP rate into `paVs` / `gidpRate` before the watcher's per-PA loop runs. The legacy `pReach` + 2-state DP path in `lib/prob/{reach-prob,inning-dp}.ts` is retained for back-compat but is **not invoked by the watcher**. See `docs/PROBABILITY_MODEL.md`.
- **Aging adjustment (v2.2).** `lib/mlb/aging.ts:applyAging` projects prior-season rates forward 1 year inside `combineSeasonsPa`. Role-specific peaks (batter 27, pitcher 28) with per-outcome slopes; HR fades past peak, K rises for batters and falls for pitchers. Each multiplier clamped `[0.85, 1.15]`. Unknown age (missing `currentAge` + `birthDate`) → identity. Don't disable the aging step under "the data is noisy" rationale — Marcel / Steamer / ZiPS / THE BAT all carry an explicit aging step; age-blind blending was the single biggest published-projection gap our model had.
- **Smooth TTOP (v2.2).** `lib/prob/ttop.ts` uses linear slopes (`K -0.0040/PA`, `BB +0.0030/PA`, `HR +0.0110/PA`) clamped to `K ∈ [0.85, 1.0], BB ∈ [1.0, 1.15], HR ∈ [1.0, 1.45]`. Slopes calibrated to land on the published Lichtman bucket midpoints (PA 13 / 22 / 31). Don't revert to step-bucketed factors — Brill/Deshpande/Wyner (JQAS 2023) showed the step at pass 3 is largely a compositional artifact. `ttoIndex(pa)` is retained for log lines but no longer load-bearing on the probability path.
- **xHR denoiser is hitter-only (v2.2).** `lib/env/expected-stats.ts:hrRateMultiplier` is applied to batter HR rates with `n0 = 50 BBE` EB shrinkage and clamp `[0.7, 1.3]`. Same multiplier on both L and R sides (Savant doesn't publish handedness-split expected stats). **Don't apply to pitchers** — BP "Siren Song" shows pitcher xstats barely correlate year-over-year; the pitcher-side analogue would be noise injection. Scrape: `https://baseballsavant.mlb.com/leaderboard/expected_statistics?type=batter&year=YYYY&min=10` (cache 24h, degrades to identity on failure).
- **Per-handedness park factors (v2.2).** `getParkComponentFactors` fetches `batSide=L` + `batSide=R` + combined in parallel. Combined is the fallback for either side. Don't collapse back to a single-index scrape — HR factors swing >20% by handedness in extreme parks (Yankees RF, Fenway LF).
- **Reliever 7-day workload drag (v2.2).** `lib/env/workload.ts:workloadKFactor` ramps K-rate multiplier from 1.00 at 120 pitches to 0.97 at 200 pitches. Don't widen the band — Driveline / FanGraphs B2B research bounds the effect tightly; widening reintroduces noise. Cache TTL 6h via `k.workload(playerId)`.
- **Stuff+/Pitching+ pitcher-quality prior (v2.2).** `lib/env/stuff.ts:stuffFactors` maps `(Pitching+ − 100)` to K↑ slope `+0.0015` and HR↓ slope `-0.002`, clamped `[0.95, 1.05]`. Join to MLB Stats API ids via FanGraphs `xmlbamid` — degrades to identity when the row is missing or the scrape fails. Don't widen the clamp; Stuff+ is a small denoiser, not a replacement for per-PA splits.
- **Per-batter GIDP rate (v2.2).** `BatterPaProfile.gidpRate` is EB-shrunk against the league rate (0.10) with `n0 = 250 PA`, clamped `[0.03, 0.20]`. Threaded into Markov via `pAtLeastOneRun(start, lineup, { gidpRates })`. Falls back to 0.10 league mean if the array or entry is missing. The GIDP/PA → GIDP/eligible-ipOut bridge multiplier is `LEAGUE_GIDP_RATE / LEAGUE_GIDP_PER_PA ≈ 6.67` — keep that constant if you tune the league mean.
- **Stratified calibration shim (v2.2).** `lib/prob/calibration.ts` lookup chain: `"${bucket}-${half}"` → bucket only → `"global"` → identity. Buckets are `"1"` (inning 1), `"2-6"`, `"7-9"`, `"10+"` (extras with Manfred runner). Ship the lookup wiring even while every bin is identity — the post-1k-sample swap is a one-line change. Both watcher call sites (live recompute + opposite-half pre-compute for full-inning composition) pass `inning` + `half`. `loadCalibrator(table | map | null)` accepts a single table (used as global default), a per-bucket map, or null.
- **ABS-tightened framing clamp (v2.2).** `lib/env/framing.ts` coefficients halved (`K_MULT_PER_RATE = 5.0`, `BB_MULT_PER_RATE = -4.0`), default clamp tightened to `[0.97, 1.03]` (was `[0.95, 1.05]`). Reflects the 2026 ABS challenge regime — walk rate up to 9.6% YTD vs 8.4% in 2025. `NRXI_FRAMING_CLAMP=0.02` env override available for mid-season retuning. `NRXI_DISABLE_FRAMING=1` kill switch retained for full-ABS adoption.
- **Switch-hitter rule:** `actual` (canonical platoon advantage) by default — switch hitters bat opposite the pitcher's throwing hand. Legacy v1 `max(L, R)` reachable via `NRXI_SWITCH_HITTER_RULE=max`.
- **nrXi definition:** v2 computes `P(nrXi) = 1 − P(≥1 run scores)` directly via the Markov chain — no proxy. The legacy `pHitEvent` field name on `NrXiResult` is preserved for UI back-compat but its semantics are now exact.
- **Decision moment (half-inning):** `outs === 3` OR `inningState ∈ {middle, end}` OR `(half === "Top" && outs === 0)`. Fires at every half-boundary.
- **Decision moment (full-inning):** half-inning predicate AND `upcoming.half === "Top"`. Fires only when the next batter will lead off a TOP half (= a new inning is approaching). That covers end-of-bottom (upcoming flips to Top of N+1), inningState="end" between innings, and start of any inning's top (mid-game leadoff or game start). Does NOT fire at top→bottom mid-inning flips. **Why upcoming.half (not raw `state.half` + flag combos):** MLB's live feed inconsistently uses inningState="middle" vs "end" at boundaries, sometimes advances `inning`/`isTopInning` before posting an end state. `upcoming.half` from `getUpcomingForCurrentInning` is the canonical "next half to bat" and is the only reliable signal.
- **Full-inning composition:** `(rest_of_top × clean_bottom)` mid-top → `clean_bottom` at top-end (= half-inning) → `rest_of_bottom` mid-bottom → `clean_top × clean_bottom` of next inning at bottom-end. **9th inning is conditionally top-only** — predicate `services/full-inning.ts:shouldSkipBottomNinth({inning, half, homeRuns, awayRuns})` returns true only at top-9 *and* home is currently leading (bottom-9 won't be played). Tied or visitors-ahead in top-9 → bottom-9 plays and we compose normally. Bottom of 9 + extras always compose normally.
- **Manfred runner (extras):** at any half-boundary in inning ≥ 10, `readMarkovStartState` returns `{outs: 0, bases: 2}` (runner on 2B). Mid-PA we trust the live feed's offense state — MLB Stats API populates the Manfred runner there at extra-half leadoff. The `oppHalfCleanCache` also seeds `bases: 2` when `upcoming.inning >= 10`. Per-inning capture accepts both `bases === 0` (regulation) and `bases === 2` (Manfred) as clean half-boundary state.
- **Break-even rounding:** American odds rounded to nearest 5 in display; raw value used for EV calc.
- **League-rate constants** (`LEAGUE_PA` in `lib/mlb/splits.ts`) are 2024–2025 averages by pitcher hand. Refresh annually.
- **Empirical-Bayes shrinkage** prior strength is **role-specific**: hitter `n0 = 200` PA, pitcher `n0 = 500` PA. Both apply against **true** (unweighted) PA so the calibration stays grounded in real sample size. The hitter/pitcher split is intentional — pitcher batted-ball-driven components (BABIP ≈ 2000 BIP, HR ≈ 1320 BF to stabilize) need a much larger prior than hitter equivalents (820 BIP, 170 PA). 500 is a single-knob compromise across the per-PA outcome bundle (K%/BB% would prefer smaller, BABIP/HR much larger). Don't collapse hitter/pitcher back to one shared n0 without a calibration study. Constants live in `lib/mlb/splits.ts:BATTER_BLEND` / `PITCHER_BLEND`.
- **Cross-season blend (v2):** **role-specific** Marcel-style multipliers on per-PA blending of current + prior regular-season splits. Hitter `wCurrent:wPrior = 3:2` (decay 0.67); pitcher `2:1` (decay 0.5). The pitcher/hitter split mirrors Marcel's own (5/4/3 hitters vs 4/3/2 pitchers) — pitcher rates are noisier year-to-year, so the prior year carries less weight. The multiplier biases *which* observations dominate; shrinkage strength stays tied to actual PA. Strict-Marcel form (using `wCurrent·current + wPrior·prior` as `n` for shrinkage) would silently distort the effective shrinkage strength — would need a calibration retune.
- **Within-season recency blend:** `recentWeight = 0.10` of the final blended rate carried by the L30 sample, applied only when L30 PA ≥ `recentMinPa = 80`. Same values for hitters and pitchers. Tightened from the previous (0.30 weight at PA ≥ 20) to better match published practice — *The Book* and empirical replications (Razzball L3-L5, FiveThirtyEight hot-hand) put the L30 signal at ~5 wOBA points / 0.0-0.2% improvement, and **no major published projection system uses an L30 component at all**. Keeping a small blend is a deliberate in-game-model deviation; widening it back to 30% reintroduces noise. Constants in `lib/mlb/splits.ts:BATTER_BLEND` / `PITCHER_BLEND`.
- **TTOP factors** (`lib/prob/ttop.ts`) come from Tango / Lichtman / Carleton published values. Don't tune without backtest data.
- **Calibration shim is identity in v1.** Fit isotonic regression from production `(predicted, actual)` pairs once ≥1k inning outcomes accumulate, then load via `loadCalibrator(table)`.
- **v2.1 framing + OAA:** EB shrinkage priors `n0 = 2000` called pitches (framing), `n0 = 200` opportunities (OAA). Factor clamps: framing `[0.95, 1.05]`, defense `[0.90, 1.10]`. Both default to identity when scrape fails or live alignment is missing — pipeline degrades gracefully to v2.
- **Robo-ump kill switch:** `NRXI_DISABLE_FRAMING=1` zeroes the framing effect. Flip when MLB's ABS challenge system goes full-season.
- **Live defensive alignment** read from `liveData.linescore.defense.{catcher, first, second, third, shortstop, left, center, right}` ids each tick. The watcher's `defenseAlignmentKey` is part of the recompute trigger so defensive subs auto-invalidate the cache.
- **User-facing settings defaults:** `predictMode: "full"`, `viewMode: "single"` (`lib/hooks/use-settings.tsx`). Both defaults are the LEFT option of their toggle. Changing the defaults is a UX call — be deliberate.

## Validator hook quirks (advisory only)

The session has a `posttooluse-validate` hook that flags things like "Workflow files should import and use logging." It runs a regex against specific lines and frequently misses logging that's actually present. **Treat its suggestions as advisory.** Don't add redundant `console.log` just to silence it. The real signals are: TypeScript errors, build errors, and test failures.
